Reference

Documentation on OIDC connectivity, scopes, the authorisation code flow, and data handling.

{# ── Connectivity ── #}

This tool involves three parties. All of the paths below must be reachable for login to work.

👤 Browser
→ GET /, /login, /claims (HTTP/S to this app)
→ Redirected to /authorize on the SSO server (login UI)
→ Redirected back to /callback on this app (with auth code)
💻 This App (server)
→ Fetches /.well-known/openid-configuration (on startup and login)
→ Exchanges auth code for tokens at token_endpoint (back-channel)
→ Fetches jwks_uri to verify token signatures
→ Fetches userinfo_endpoint for profile claims
🔒 SSO Server (Keycloak / Kanidm / etc.)
Common issue: The browser can reach the SSO login page, but the app's back-channel token exchange fails. This happens when the container can't resolve the SSO server hostname (DNS differs inside Docker). Use the connectivity check on the home page to diagnose.

Logout

RP-initiated logout sends the browser to the SSO server's end_session_endpoint with the id_token_hint and a post_logout_redirect_uri. Keycloak supports this; Kanidm currently does not (local-only session clear happens instead). Register {{ request.host_url }} as an allowed post-logout URI in your client config.

{# ── Scopes & Claims ── #}

The claims you receive depend entirely on which scopes you request. Add scopes to OIDC_SCOPE in your .env file. You must also grant the scopes in your provider's client configuration.

ScopeClaims providedNotes
openidsub, iss, aud, exp, iatRequired — activates OIDC mode
emailemail, email_verifiedStandard OIDC
profilename, given_name, family_name, preferred_username, picture, locale, zoneinfo, updated_atStandard OIDC
addressaddressStandard OIDC — rarely used
phonephone_number, phone_number_verifiedStandard OIDC — rarely used
offline_access(no new claims)Requests a refresh token; provider must allow it
rolesrealm_access, resource_accessKeycloak-specific
groupsgroupsProvider-specific — varies by server
Want more claims? You can't add scopes mid-session — OIDC scopes are set at login time. To get additional claims, add the scopes to OIDC_SCOPE and sign in again. The server only returns claims for scopes it has granted, so you may also need to configure the scope in your provider's client/consent settings.

The code_verifier never leaves the app server, so intercepting the auth code is useless without it — this is the security benefit of PKCE. Token signing (ES256/RS256) lets the app verify tokens were issued by the expected server without a back-channel call.

{# ── Federated OIDC Flow ── #}

In a federated identity scenario your local SSO server does not hold user accounts directly — instead it acts as a proxy identity provider, forwarding authentication requests to one or more upstream identity federations. From this application's perspective the flow looks identical to a direct OIDC login; the extra hops happen invisibly behind the local SSO server.

Parties involved

👤 Browser
→ Sends user to /authorize on the local SSO server
→ Local SSO shows a provider-selection screen (if multiple federations are configured)
→ Redirected to the upstream federation hub to authenticate
→ Redirected back to local SSO with a federation assertion
→ Local SSO exchanges the assertion for its own tokens, redirects to /callback
💻 This App (service provider)
→ Receives tokens issued by the local SSO server — not by the upstream federation
→ iss claim will be the local SSO server's issuer URL
→ sub may be a local identifier, a scoped identifier, or a federation persistent ID depending on configuration
🔒 Local SSO Server (proxy IdP)
→ Authenticates outbound via SAML 2.0 or OIDC to the federation hub
→ Maps federation attributes to local OIDC claims
🌐 Upstream Federation Hub (eduGAIN, national research & education network, etc.)

What this means for your tokens

ClaimTypical behaviour in federated login
iss Always the local SSO server — never the upstream federation. The proxy rewrites the issuer.
sub May be a local UUID, a scoped persistent ID (e.g. user@home-institution.example), or a federation-assigned ePTID depending on attribute mapping configuration.
email Sourced from the home institution via the federation. May be absent if the home institution does not release it.
acr Authentication context from the upstream IdP, translated by the local SSO. May reflect MFA level or a federation-defined assurance profile (e.g. Assurance Profile Low/High).
Federation attributes Home-institution attributes (affiliation, entitlements, schac fields) are mapped to OIDC claims by the local SSO. Which claims appear depends entirely on the attribute-release policy at the home institution and the local SSO's mapper configuration.

Common diagnostic issues

Missing claims after federated login? The home institution controls which attributes it releases to the federation. If email, given_name, or other expected claims are absent, the attribute-release policy at the home institution needs to be updated — this cannot be fixed at the local SSO or service-provider level.
Unexpected sub value? Federated logins often produce a pairwise or persistent subject identifier derived from the user's home identity rather than a local account ID. This is by design and ensures the service provider cannot correlate the user across different federation members.
Callback URL registration. The local SSO server's OIDC client registration must include this tool's /callback URL. For multi-provider deployments, each provider has its own callback path (/callback/<provider_id>). Check the OIDC Flow tab for the full callback sequence.
{# ── Data & Privacy ── #}

What this tool stores

Sessions use client-side signed cookies — token data lives in your browser, not in a server-side database. The server holds tokens in memory only for the brief duration of processing a request. Nothing is written to disk, a database, or any external service.

What the operator's server processes

The server code receives and decodes your tokens in order to render them for display. It does not log token values. With FLASK_DEBUG=true Flask may surface token data in error traces — a properly configured instance keeps debug mode off.

Your password is never seen by this tool — you authenticate directly on your OIDC provider's login page.

Third parties

This tool does not connect to any analytics, advertising, or tracking service. The only outbound network calls it makes are to your configured OIDC provider.

Trust the operator. As with any OAuth2 client, you are trusting the person who runs this instance. Only sign in with accounts whose data you are comfortable sharing with the instance operator. Use the minimum scopes you need (openid email profile is sufficient for most diagnostic purposes).
