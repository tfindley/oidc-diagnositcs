Reference

Documentation on OIDC connectivity, scopes, the authorisation code flow, and identity brokering.

{# ── Connectivity ── #}

This tool involves three parties. All of the paths below must be reachable for login to work.

👤 Browser
→ GET /, /login, /claims (HTTP/S to this app)
→ Redirected to /authorize on the SSO server (login UI)
→ Redirected back to /callback on this app (with auth code)
💻 This App (server)
→ Fetches /.well-known/openid-configuration (on startup and login)
→ Exchanges auth code for tokens at token_endpoint (back-channel)
→ Fetches jwks_uri to verify token signatures
→ Fetches userinfo_endpoint for profile claims
🔒 SSO Server (Keycloak / Kanidm / etc.)
Common issue: The browser can reach the SSO login page, but the app's back-channel token exchange fails. This happens when the container can't resolve the SSO server hostname (DNS differs inside Docker). Use the connectivity check on the home page to diagnose.

Logout

RP-initiated logout sends the browser to the SSO server's end_session_endpoint with the id_token_hint and a post_logout_redirect_uri. Keycloak supports this; Kanidm currently does not (local-only session clear happens instead). Register {{ request.host_url }} as an allowed post-logout URI in your client config.

{# ── Scopes & Claims ── #}

The claims you receive depend entirely on which scopes you request. Add scopes to OIDC_SCOPE in your .env file. You must also grant the scopes in your provider's client configuration.

ScopeClaims providedNotes
openidsub, iss, aud, exp, iatRequired — activates OIDC mode
emailemail, email_verifiedStandard OIDC
profilename, given_name, family_name, preferred_username, picture, locale, zoneinfo, updated_atStandard OIDC
addressaddressStandard OIDC — rarely used
phonephone_number, phone_number_verifiedStandard OIDC — rarely used
offline_access(no new claims)Requests a refresh token; provider must allow it
rolesrealm_access, resource_accessKeycloak-specific
groupsgroupsProvider-specific — varies by server
Want more claims? You can't add scopes mid-session — OIDC scopes are set at login time. To get additional claims, add the scopes to OIDC_SCOPE and sign in again. The server only returns claims for scopes it has granted, so you may also need to configure the scope in your provider's client/consent settings.

The code_verifier never leaves the app server, so intercepting the auth code is useless without it — this is the security benefit of PKCE. Token signing (ES256/RS256) lets the app verify tokens were issued by the expected server without a back-channel call.

{# ── Identity Brokering ── #}

Identity brokering is when your SSO server (e.g. Keycloak) does not authenticate the user directly — instead it delegates to an upstream identity provider such as Microsoft Entra ID, Google, GitHub, or another OIDC/SAML server. Keycloak acts as the broker: it handles the upstream authentication, maps the upstream claims to its own local claims, then issues its own tokens to this application. From this app's perspective the flow looks like a normal OIDC login; the upstream hops happen entirely behind Keycloak.

What this means for your tokens

ClaimBehaviour with identity brokering
iss Always Keycloak's issuer URL — never the upstream IdP. The broker rewrites the issuer before issuing tokens to this app.
sub A Keycloak-assigned identifier for the brokered user — not the upstream IdP's subject. Keycloak may create a local account linked to the upstream identity, or use a derived persistent ID depending on broker configuration.
email, name Sourced from the upstream IdP via Keycloak's claim mappers. May be absent if the upstream IdP does not release them or the mapper is not configured.
acr Authentication context as reported by Keycloak. May reflect the upstream IdP's MFA level if Keycloak is configured to pass it through.
Upstream attributes Any upstream IdP claims you want visible here must be explicitly mapped in Keycloak's identity provider mapper configuration. Unmapped claims are silently discarded.

Common diagnostic issues

Missing claims after brokered login? Check two places: (1) whether the upstream IdP is releasing the claim to Keycloak, and (2) whether Keycloak has an identity provider mapper configured to forward it into the local token. Both must be in place — a claim released by the upstream IdP but not mapped in Keycloak will not appear here.
Unexpected sub value? Keycloak assigns its own local subject identifier to brokered users. The upstream IdP's subject is not directly exposed unless you explicitly add an identity provider attribute mapper for it.
Callback URL registration. Keycloak's OIDC client for this app must include this tool's /callback URL as a valid redirect URI. For multi-provider deployments, each provider has its own path (/callback/<provider_id>). Check the OIDC Flow tab for the full callback sequence.
