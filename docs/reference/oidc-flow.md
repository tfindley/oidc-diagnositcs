# OIDC Authorization Code Flow

This tool implements the **Authorization Code flow with PKCE** (Proof Key for Code Exchange) as defined in [RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749) and [RFC 7636](https://datatracker.ietf.org/doc/html/rfc7636).

## Sequence diagram

```
Browser                    This App                   SSO Server
  │                            │                            │
  │  GET /login                │                            │
  │ ──────────────────────────►│                            │
  │                            │  Generate code_verifier    │
  │                            │  (PKCE S256)               │
  │                            │                            │
  │  302 → /authorize          │                            │
  │◄───────────────────────────│                            │
  │                            │                            │
  │  GET /authorize?client_id  │                            │
  │  &code_challenge=...       │                            │
  │ ───────────────────────────────────────────────────────►│
  │                            │                            │  Show login UI
  │  User authenticates        │                            │◄──────────────
  │ ◄───────────────────────────────────────────────────────│
  │                            │                            │
  │  302 → /callback?code=...  │                            │
  │◄───────────────────────────────────────────────────────-│
  │                            │                            │
  │  GET /callback?code=...    │                            │
  │ ──────────────────────────►│                            │
  │                            │  POST /token               │
  │                            │  {code, code_verifier}     │
  │                            │ ──────────────────────────►│
  │                            │                            │  Verify PKCE
  │                            │  {access_token, id_token,  │  Sign tokens
  │                            │   refresh_token, userinfo} │◄──────────────
  │                            │◄───────────────────────────│
  │                            │                            │
  │                            │  GET /userinfo             │
  │                            │ ──────────────────────────►│
  │                            │◄───────────────────────────│
  │                            │                            │
  │  302 → /claims             │  Store tokens in session   │
  │◄───────────────────────────│                            │
```

## Why PKCE?

The `code_verifier` is a random secret generated by the app at login time. Its hash (`code_challenge`) is sent to the SSO server with the authorization request. When the app exchanges the auth code for tokens, it must provide the original `code_verifier`.

This means that even if an attacker intercepts the auth code (e.g. from a redirect URI or server logs), they cannot exchange it for tokens without the `code_verifier` — which never leaves the app server.

PKCE S256 is the default and is required by Kanidm. It is recommended for all deployments.

## Token signing

Tokens are signed by the SSO server using ES256 or RS256 (configurable via `OIDC_TOKEN_SIGNING_ALG`). The app fetches the SSO server's public keys from `jwks_uri` to verify token signatures. The JWT Decoder page also supports manual JWKS verification via the browser's Web Crypto API.

## Configuration

| Variable | Default | Description |
| --- | --- | --- |
| `OIDC_PKCE_METHOD` | `S256` | PKCE method: `S256`, `plain`, or `disabled` |
| `OIDC_TOKEN_SIGNING_ALG` | *(auto)* | Enforce `ES256` or `RS256`; unset = accept server default |
